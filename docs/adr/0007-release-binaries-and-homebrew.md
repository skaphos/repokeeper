# ADR-0007: Release Binary Publishing and Homebrew Distribution

**Status:** Accepted
**Date:** 2026-04-18
**Author:** Shawn Stratton

## Context

The `v0.7.0` (and `v0.6.1`) release jobs failed in the `GoReleaser` step. Goreleaser successfully built, archived, SBOM'd, and signed all artifacts, but every upload to the GitHub release failed with:

> `422 Cannot upload assets to an immutable release`

Confirmed via `gh api repos/skaphos/repokeeper/releases/tags/v0.7.0`: the release exists, is marked `immutable: true`, and has zero assets. Users have no way to download prebuilt binaries from the GitHub release page.

### Root cause

`googleapis/release-please-action@v4` creates the GitHub release itself when it merges its release PR. The repository (or org) has GitHub's new "immutable releases" enabled, which causes release-please to create the release with `immutable: true`. Immutable releases reject all subsequent asset uploads and edits.

When the tag push then triggers `.github/workflows/release.yml`, goreleaser detects the existing release and tries to upload its artifacts — all uploads return 422, and the workflow fails *after* signing, SBOM generation, and Homebrew cask rendering have already run.

### Secondary problem: Homebrew distribution

`.goreleaser.yaml` uses `homebrew_casks:` to publish to `skaphos/homebrew-tools`. Two things are wrong with the current setup — but not what I originally suspected:

1. **Our cask is stale at 0.6.0.** `homebrew-tools/Casks/repokeeper.rb` is still pinned at `0.6.0` because the `0.6.1` and `0.7.0` goreleaser runs failed before the cask update step ran. This is a symptom of the primary issue (the immutable-release upload failure); once goreleaser owns the release end-to-end, tap updates will flow normally.
2. **The cask has no Gatekeeper workaround.** On Apple Silicon, installing an unsigned binary via a cask attaches `com.apple.quarantine`, which makes macOS Gatekeeper refuse to run it ("cannot be opened because the developer cannot be verified"). The fix is a `postflight` hook that strips the xattr at install time; see Section 2 of the Decision for the choice of hook over the older `no_quarantine true` cask DSL stanza and why that matters on macOS 13+.

**Not switching to `brews:`.** My first instinct was to switch to a formula, since Homebrew's own convention for unsigned CLI binaries is a formula. But running `goreleaser check` emits:

> `brews is being phased out in favor of homebrew_casks, check https://goreleaser.com/deprecations#brews for more info`

Goreleaser is deprecating `brews:` entirely. Fighting that tide for a repo that already publishes a cask isn't worth it; the cask just needs to be configured correctly.

## Decision

### Supported runtimes

GoReleaser continues to build prebuilt binaries for the full matrix of compute runtimes our users target:

- `linux/amd64`, `linux/arm64`
- `darwin/amd64`, `darwin/arm64`
- `windows/amd64`, `windows/arm64`

This matches the `goos` / `goarch` matrix already declared under `builds[0]` in `.goreleaser.yaml`. We are not adding 32-bit x86, freebsd, netbsd, or openbsd targets (no observed demand), and we are not introducing `universal_binaries:` for darwin — the separate arm64 and amd64 archives are cheaper to mirror into Homebrew casks and CI runners than a fat binary.

### 1. Goreleaser owns the GitHub release and release body; release-please bumps version; the `release-please.yml` workflow pushes the tag

Release-please stays the trigger — it still opens the release PR, bumps `.release-please-manifest.json`, and writes `CHANGELOG.md` in the repo for humans browsing the source. It stops creating the GitHub release object entirely; the release notes are generated by goreleaser from git history between tags.

**`release-please-config.json`** — tell release-please to skip the GitHub release. Keep `CHANGELOG.md` generation in-repo; that file is still useful as a browsable history even though it's no longer the source of the release notes.

```json
"packages": {
  ".": {
    "release-type": "go",
    "skip-github-release": true
  }
}
```

**Tag creation caveat.** Under `skip-github-release: true`, release-please-action v4 does **not** push the `vX.Y.Z` tag. In its model the tag is created via the GitHub Releases API as part of the release object, and skipping the release skips the tag. We verified this empirically: after merging PR #190 to release 0.7.1, `.release-please-manifest.json` and `CHANGELOG.md` were updated on `main` but no `v0.7.1` tag was ever pushed, so `release.yml` never fired.

**`.github/workflows/release-please.yml`** therefore takes over the tag push. After the release-please-action step runs, the workflow checks out `main`, reads the version from `.release-please-manifest.json`, and — if no `v${VERSION}` tag exists — creates and pushes it using the same bot token that merged the release PR. The step is idempotent (no-op when the tag already exists) and self-healing: if any release-please run ever dies after updating the manifest but before pushing the tag, the next push to `main` re-runs the step and pushes the missing tag.

This leaves release-please-action responsible for:
- opening / merging the release PR
- bumping `.release-please-manifest.json`
- appending the new section to `CHANGELOG.md` (in-repo only)

…and the `release-please.yml` workflow pushes the `vX.Y.Z` tag that triggers `release.yml`.

**`.goreleaser.yaml`** — add an explicit `release:` block so goreleaser creates and publishes the GitHub release end-to-end:

```yaml
release:
  mode: replace     # fresh release, goreleaser owns the body
  draft: false
  make_latest: true
```

The existing top-level `changelog:` block stays and becomes the source of truth for GitHub release notes. It already filters out `docs:`, `test:`, and `ci:` commits; we extend the exclude list with `^chore:` and `^chore\(` so dependency bumps and repository housekeeping do not clutter the user-facing release notes — those commits remain visible in `git log` and in `CHANGELOG.md` for anyone who wants the full history. We also add commit groups (Features, Bug Fixes, Performance Improvements, Others) so the rendered release body matches the conventional-commit categories release-please uses in `CHANGELOG.md`.

**`.github/workflows/release.yml`** — no extraction step. Goreleaser already diffs commits between the previous tag and the current tag when generating its changelog, so as long as the workflow checks out with `fetch-depth: 0` (it does) and tags are fetched, this works out of the box. The only change to the workflow is cosmetic — the `goreleaser_args` output stays as-is.

**Why this shape:**
- Single source of truth for the GitHub release body: goreleaser's `changelog:` block.
- No fragile text extraction from a Markdown file.
- `CHANGELOG.md` in the repo is independently maintained by release-please for humans reading the source tree, and drifts from the GitHub release body only in filtering rules — which is acceptable, since the two audiences differ.
- Goreleaser owns release creation, so "immutable release" can never collide with asset uploads — goreleaser's CREATE call sets the body it wants from the start.

**Alternative considered:** parse `CHANGELOG.md` and pass it to goreleaser via `--release-notes`. Rejected — the awk/sed extraction is fragile, and keeping two changelog generators (release-please for the repo, goreleaser for the release) is simpler than plumbing one into the other.

**Alternatives considered at the pipeline level:**

- **Drop release-please; let goreleaser drive everything.** A survey of active Go CLIs (cli/cli, goreleaser, anchore/syft + grype, sigstore/cosign, bufbuild/buf, lazygit, k9s, fluxcd/flux2, grafana/k6, charmbracelet/{gum,glow,vhs}) finds that the overwhelming majority skip release-please entirely and let goreleaser own the release using its `changelog:` block with `use: github-native` or grouped conventional-commit templates, triggered by manual `workflow_dispatch` or a `push: tags: v*` event. This is the *common* path among Go CLIs. Rejected for now because release-please gives us two things we actively want: (a) a PR-review gate where a human approves the version bump and accumulated changelog before the tag is cut, and (b) automatically-maintained `CHANGELOG.md` in the repo. A goreleaser-only flow can preserve the gate by triggering on `workflow_dispatch` rather than on every push, but it gives up the in-repo changelog. If release-please's API surface keeps shifting or we trim dependencies after notarization lands, migrating to goreleaser-only with a `workflow_dispatch` trigger is the natural simplification and has no technical blockers.
- **Replace release-please with `svu` + goreleaser.** `svu` computes the next semver from conventional commits and pushes the tag on every push to `main`. **Rejected.** That cadence would release on every merge; we want intentional, human-gated release batches (multiple merges accumulated into one version bump that a maintainer approves via the release-please PR before it lands). `svu` cannot provide that gate; release-please's PR flow does. Any future replacement for release-please must preserve the gate, not eliminate it.
- **Relax the org-level immutable-releases enforcement.** Rejected. Immutable releases are a supply-chain guardrail; working around a tooling gap by removing a security control is the wrong trade. The feature-request to add "tag-only without release" support to release-please-action exists upstream ([release-please-action#1034](https://github.com/googleapis/release-please-action/issues/1034), open since Sep 2024) and will eventually land; our workflow-level tag push is the interim.

Two real OSS projects run essentially the same shape we are adopting (release-please with `skip-github-release: true`, plus a workflow step that pushes the tag explicitly before goreleaser runs): `paperclipinc/paperclip-operator` and `openclaw-rocks/openclaw-operator`. Both cite the immutable-releases conflict in inline comments on the workflow. So we are on a narrow but validated path.

### 2. Keep `homebrew_casks:`, but make it install cleanly on Apple Silicon

Update the existing `homebrew_casks:` entry in `.goreleaser.yaml` to bypass Gatekeeper and generate shell completions:

```yaml
homebrew_casks:
  - repository:
      owner: skaphos
      name: homebrew-tools
      branch: main
      token: "{{ .Env.HOMEBREW_TAP_GITHUB_TOKEN }}"
    homepage: "https://github.com/skaphos/repokeeper"
    description: "Track and manage many Git repositories from a single config"
    license: "MIT"
    binaries:
      - repokeeper
    custom_block: |
      postflight do
        on_macos do
          system_command "/usr/bin/xattr",
                         args: ["-dr", "com.apple.quarantine", "#{staged_path}/repokeeper"]
        end
      end
```

- `binaries: [repokeeper]` (plural list) is the current form. An earlier draft of this ADR inverted the deprecation direction: the singular `binary:` field was deprecated in goreleaser v2.12.6 in favor of the list, and our existing config already uses the correct plural form. No code change needed here beyond leaving it alone.
- **Gatekeeper: `postflight` xattr hook, not `no_quarantine true`.** Two approaches exist to make macOS Gatekeeper accept an unsigned binary installed via a Homebrew cask, and they are not equivalent on current macOS:
  - `no_quarantine true` is a one-line cask DSL stanza that tells Homebrew not to attach `com.apple.quarantine` to the downloaded artifact. It is simple and declarative, and was reliable in the Mojave / Catalina era. On macOS 13+ it is increasingly ineffective: Gatekeeper evaluates binaries against `LSQuarantine*` provenance metadata that the OS itself records when the artifact is downloaded via `curl` from a GitHub Releases URL, regardless of whether Homebrew set the cask-level xattr. Users still see the "cannot be opened because the developer cannot be verified" dialog in practice.
  - A `postflight` hook that runs `xattr -dr com.apple.quarantine "#{staged_path}/repokeeper"` strips the extended attribute on the user's machine at install time, *after* the download. This is the pattern goreleaser's current `homebrew_casks` docs recommend. It produces the same installed outcome regardless of what the OS attached at download.

  We adopt the `postflight` hook. The tradeoff is a few lines of inline Ruby that goreleaser cannot lint, versus a one-line stanza that may no-op at the OS level. We accept the extra lines because the hook actually achieves the goal on the macOS versions our users run today. The correct long-term fix is to notarize the binary with an Apple Developer ID so Gatekeeper accepts it without any workaround — out of scope for this ADR, to be revisited once the project owns a signing/notarization pipeline.

  **Linuxbrew guard.** Homebrew casks are nominally macOS-only, but Linuxbrew will install them and then run the `postflight` hook. `/usr/bin/xattr` does not exist on Linux, so an unguarded `system_command` exits 127 and Linuxbrew rolls back the upgrade (observed on `v0.7.1`). We therefore wrap the `system_command` in an `on_macos do ... end` block so the hook is a no-op on Linuxbrew and the install succeeds. The xattr strip is irrelevant off macOS — there is no Gatekeeper to bypass — so this is a pure platform guard, not a behaviour change.

Shell completions via `generate_completions_from_executable` require the binary to support a `completion` subcommand; I'll check that separately and add it in a follow-up PR if it's wired up.

No changes needed in `../homebrew-tools/`: the existing `Casks/repokeeper.rb` will be overwritten by goreleaser on the next successful release.

### 3. Backfill v0.7.0 assets (one-off)

Because `v0.7.0` exists as an immutable release with no binaries, no future workflow run can attach artifacts to it. We have two options:

- **A.** Leave `v0.7.0` asset-less. Users must wait until `v0.7.1` / `v0.8.0` for binaries. Simple, no manual intervention.
- **B.** Delete and recreate `v0.7.0` via `gh release delete v0.7.0` + re-run `goreleaser release --clean` locally with a personal token. Risky: the tag is already public, and deleting an "immutable" release may not be supported by the API.

**Recommendation:** Option A. The fix lands in `v0.7.1` and we accept that `v0.7.0` is binary-less. Cut a patch release immediately after merging this PR to validate the end-to-end pipeline.

## Consequences

### Positive
- GitHub releases actually ship prebuilt binaries, checksums, SBOMs, and cosign signatures.
- `brew install skaphos/tools/repokeeper` works on macOS and Linux without Gatekeeper warnings.
- `CHANGELOG.md` stays authored by release-please, while GoReleaser uses that changelog content as the single source of truth for the GitHub release body.
- Goreleaser's existing `signs:`, `sboms:`, and attestation steps continue to work unchanged.

### Negative / risks
- `v0.7.0` stays asset-less; users who already found it get a release page with no downloads until `v0.7.1`.
- Until the first successful goreleaser run under the new pipeline, `skaphos/homebrew-tools/Casks/repokeeper.rb` stays pinned at `0.6.0` with no Gatekeeper workaround. `brew upgrade` will not pick up a newer version or the `postflight` hook until goreleaser overwrites the cask on `v0.7.1` (or later).
- The `postflight` block embedded via `custom_block:` is inline Ruby that goreleaser does not validate. If the cask DSL shape changes upstream in Homebrew, breakage is visible only at install time on a user's machine — worth exercising on at least one macOS host as part of the verification plan.

### Verification plan
1. Merge this PR into `main`.
2. Let release-please open its next release PR.
3. Merge the release PR → inspect the release-please workflow run → confirm release-please pushes the tag but does **not** create a GitHub release.
4. Watch the `Release` workflow run → confirm goreleaser creates the release, attaches all archives, checksums, SBOMs, signatures, and generates the GitHub release notes from its configured changelog/git history.
5. `gh release view vX.Y.Z --json assets` should list all archives.
6. `brew tap skaphos/tools && brew install repokeeper` on macOS + Linux should succeed.
7. `repokeeper --version` should print the new version.

## Implementation plan

Phased, to keep each diff reviewable:

**Phase 1 — Release pipeline fix (this PR)**
- `release-please-config.json`: add `"skip-github-release": true` to the `.` package.
- `.goreleaser.yaml`:
  - Add `release:` block (`mode: replace`, `draft: false`, `make_latest: true`).
  - Extend the existing `changelog:` block with commit groups (Features / Bug Fixes / Performance / Others) so the rendered GitHub release body matches the conventional-commit categories, and extend `filters.exclude` with `^chore:` and `^chore\(` so dependency bumps and housekeeping do not appear in the user-facing release notes.
  - Update `homebrew_casks:` `custom_block:` to embed a `postflight` hook running `xattr -dr com.apple.quarantine` on the staged binary, guarded by `on_macos do ... end` so Linuxbrew does not try to exec a macOS-only binary. Keep `binaries: [repokeeper]` as-is — it is the current plural form; the singular `binary:` field was deprecated in goreleaser v2.12.6.

**Phase 2 — Tap sanity-check (only if needed, in `skaphos/homebrew-tools`)**
